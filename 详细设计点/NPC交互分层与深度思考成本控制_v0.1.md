# NPC交互分层与深度思考成本控制 v0.1

## 核心结论

本项目不追求“每个NPC都像真人一样长时间深聊”。更合理的目标是：

**普通NPC提供世界活性，关键NPC提供深度人格。**

普通NPC应当用低成本方式表现出生活感、记忆感和事件敏感度；深度思考则作为稀有体验、特殊任务触发器和高价值剧情节点使用。这样既能保持玩家的新鲜感和成就感，也能避免在MMO规模下被计算量拖垮。

## 交互分层

### 1. 轻交互层

适用对象：

- 普通NPC
- 高频问候、交易、求助、提醒
- 玩家没有足够信任关系时的普通沟通
- 大量玩家同时访问同一个NPC时的高并发入口

实现方式：

- 记忆摘要
- 关键词和主题提示
- 事件敏感度
- 关系值
- 职业、地点、阵营规则
- 伪思考模板

这一层的目标不是让NPC真正深思，而是让玩家感到NPC会根据环境和记忆做出差异化反应。

### 2. 可信度判断层

玩家输入不是事实，而是“信息声明”。

NPC需要先判断：

- 说话者是谁
- 与说话者的关系如何
- 这句话是否符合自身记忆
- 这句话是否涉及危险、利益或阵营
- 自己的性格是否谨慎、好奇、勇敢
- 该信息是否值得进一步调查

只有通过可信度判断后，玩家的话才可能被写入记忆、转成候选事件、触发调查任务或升级到深度思考。

### 3. 深度思考层

深度思考只应服务于高价值场景：

- 关键NPC
- 特殊任务
- 高信任关系
- 城市危机
- 阵营冲突
- 伪装、欺骗、误判等复杂情报
- 玩家掌握关键证据后的重要交涉

深度思考不应被设计成普通聊天的默认成本，而应是一种稀有资源。玩家应当感到：

**我终于让这个重要NPC认真听我说话了。**

### 4. 世界回写层

NPC的思考结果不能无限放大。深度思考的输出必须经过系统规则层约束，再写回世界。

允许写回：

- 记忆
- 关系
- 任务
- 事件
- 传闻
- 阵营态度

不允许写回：

- 绕过规则层直接改变世界事实
- 无限传播消息
- 让单次对话触发全城NPC重算
- 让NPC随意中断所有任务

## 玩家话语的最小处理链路

第一版先采用以下链路：

```text
玩家输入一句话
  -> 后端接收为 player_utterance
  -> 根据关系、NPC性格、关键词估算可信度
  -> 写入目标NPC的 message_queue
  -> fallback/thought 只把它当作倾向信息
  -> ActionPlanner 决定是否排队、调查、帮助、警戒或忽略
```

示例：

```text
玩家对护卫说：“村口来了一个很奇怪的商人。”
```

系统不直接认定“伪装怪物进入村庄”，而是形成：

```text
message_type: player_utterance
content: 村口来了一个很奇怪的商人
topic_hint: suspicious_arrival
credibility: medium
possible_action: remember_and_watch / investigate
```

护卫可能提高警惕，猎人可能想调查，商人可能无视，关键NPC则可能触发深度思考。

## 事件库设计原则

事件库不应无限枚举具体事件，而应使用“粗类型 + 结构化payload”。

例如：

```text
event_type: monster_invasion
payload:
  monsters:
    - monster_type: wolf
      count: 3
      threat_level: low
    - monster_type: goblin_shaman
      count: 1
      threat_level: medium
```

这样可以保留具体场景差异，又避免事件类型爆炸。

## 伪装怪物场景的意义

“怪物假装人类进入村庄”是本项目区别于普通NPC系统的关键证明场景。

普通系统通常只有两种处理方式：

- 系统知道它是怪物，NPC也直接按怪物处理
- 系统把它当普通人，NPC没有主观判断空间

本项目应当拆成：

- 客观真相：实体真实身份是怪物
- 可见信息：陌生人进入村庄，行为可疑
- NPC主观判断：可能是商人、间谍、怪物、骗子、无辜旅人
- 行动决策：观察、盘问、跟踪、警告、攻击、忽略

这样才能体现“NPC自己形成判断，而不是只接收已定义事实”。

## 规模控制原则

在大世界或MMO场景中，必须避免：

- 单个事件遍历所有NPC
- NPC之间对话导致消息指数级传播
- 一万个玩家同时让同一个NPC深度思考
- 普通闲聊消耗关键模型预算

推荐约束：

- 本地路由优先，只传播给相关地点、角色、阵营、关系链
- 使用 topic_id 做去重，避免同一消息被反复处理
- 普通NPC默认轻规则，关键NPC才可能深度思考
- 高频公共问题使用缓存或共享回答
- 深度思考异步化、限流化、剧情化
- 深度结果只受控写回世界，不直接引发全局重算

## Topic敏感度群组

普通NPC不应该只按“个体”接收信息，还可以按“topic敏感度层级”共享情报。

核心原则：

- 消息越重要，传播范围越广
- 同一敏感度层级内，对同一topic的基础了解可以共享
- 群体共享的是“情报版本”，不是客观真相
- 职业、阵营、地点可以进一步限制可见范围

例如“怪物可能入侵”：

- 普通村民：只知道“最近不太安全”
- 守卫和猎人：知道“村口可能有怪物活动”
- 国王、将军、斥候：知道更高敏感度版本，包含数量、方向、可信来源
- 商人：可能只知道“道路危险，贸易风险上升”

这能把消息传播从：

```text
事件 -> 遍历所有NPC -> 每个NPC单独处理
```

升级成：

```text
事件/传闻 -> topic情报池 -> 若干敏感度群组 -> 群组内NPC按需读取
```

后续职业划分也可以叠加在敏感度之上。例如国王可以只把“怪物可能入侵”的高敏感度情报开放给守卫、猎人、斥候和军官，而不是开放给所有NPC。

## 当前最小实现目标

当前代码阶段只实现：

- 玩家话语作为 `player_utterance` 进入目标NPC消息队列
- 为话语计算粗略可信度和主题提示
- 若可信度足够，生成该NPC自身的 `npc_belief`
- fallback/thought 可以把该消息当作倾向信息
- ActionPlanner 仍然负责最终任务排队和中断判断

暂不实现：

- 真模型深度思考
- 玩家话语自动生成世界事实
- 大规模topic传播系统
- topic敏感度群组共享情报池
- 伪装怪物完整信念系统

## 后续任务设计固定规则

从当前版本开始，后续新增事件和 NPC 行为时遵循以下固定设计。

### 事件驱动任务

客观事件使用“分类存储 + 职业默认流程”。

```text
world_event
  -> event_type/category/payload
  -> EventRouter 路由给相关 NPC
  -> MemorySummarizer 生成记忆
  -> EventProcessor 写入职业默认响应任务
  -> 普通 NPC 可按默认任务行动
  -> 关键 NPC 再进入 thought/LLM 决策
  -> ActionPlanner 统一决定排队、打断或忽略
```

默认任务是兜底，不是最终智能。它的职责是让普通 NPC 不沉默，并为关键 NPC 提供可解释的初始候选。

### 玩家话语驱动任务

玩家话语不能直接变成客观事件，也不能在话语处理阶段硬编码复杂社会行为。

固定链路：

```text
player_utterance
  -> DialogueInterpreter / LLM 解释话语意图
  -> topic_hint / credibility
  -> npc_reply 返回给前端作为 NPC 即时回应
  -> 目标 NPC message_queue
  -> 可信度足够时形成该 NPC 自己的 npc_belief
  -> fallback/thought/LLM 决定应对
  -> ActionPlanner 决定任务
  -> TaskExecutor 执行并回写
```

### 自由对话与事件对话固定规则

从当前版本开始，玩家与 NPC 的对话分成同一入口下的两种结果：

- 普通自由对话：模型或兜底规则生成 `npc_reply`，不生成 `npc_belief`，不改变客观世界。
- 事件相关对话：模型或兜底规则识别 `topic_hint`、`claim`、`confidence_delta`、`urgency`，可信度足够时只生成目标 NPC 自己的 `npc_belief`。

固定边界：

- 玩家话语接口必须返回 `npc_reply`，前端可直接展示，但这只是对话表现，不代表事实成立。
- `reply_text` 可以由模型生成；模型失败时允许规则兜底，但兜底不能替代长期自由对话目标。
- `dialogue_processor` 只负责解释话语、入队、生成 belief 和返回 NPC 回复，不能直接安排复杂社会行为。
- NPC 后续行动仍必须经过 `thought/fallback/LLM -> ActionPlanner -> TaskExecutor`。
- 前端头顶 `...` 只是请求等待态，用于表达 NPC 正在思考；它不写入记忆、belief 或事件。
- 目标 NPC 必须由玩家当前靠近/选中的 NPC 决定，不能固定为守卫。

例如玩家告诉商人“村口有可疑陌生人”：

```text
商人先形成 belief
  -> 商人思考/兜底规则判断安全信息应报告守卫
  -> 商人执行 report
  -> 守卫收到报告并形成自己的 belief
  -> 守卫再决定 investigate
```

禁止把这类链路写成：

```text
dialogue_processor 直接把商人的玩家话语转发给守卫
```

这种写法只能作为临时原型，不作为最终设计。

### 默认规则与大模型职责边界

默认规则：

- 负责普通 NPC 的低成本反应。
- 负责系统底层兜底。
- 负责给每类事件、每类职业提供最小可解释流程。

大模型 / 深度思考：

- 只用于关键 NPC、高价值事件、复杂欺骗、阵营冲突、重要剧情。
- 只能输出倾向、解释、候选行动、候选社交态度。
- 不能绕过 ActionPlanner 直接改变客观世界。

ActionPlanner：

- 统一裁决是否打断当前任务。
- 统一裁决任务优先级。
- 统一裁决候选行为是否能落地。

TaskExecutor：

- 只执行已被规划层接受的任务。
- 执行后再写回记忆、关系、belief 状态、世界事件或 NPC-to-NPC 消息。
